### C1. FR-002 结构化子字段的锚点形态（子标题 vs 表格 vs 代码块）

**结论**：采用二级 Markdown 子标题（如 `### 复现命令` / `### 执行结果`），不采用表格或代码块硬约束。

**依据**：
- spec 正文 §57 推荐结论已明确"沿用 `## 判定依据` 章节的双向表达骨架，硬约束证据栏为复现命令+执行输出摘录"，未要求表格。
- research §4/§5 指出现有 `NOOP_JUDGMENT_HEADING_REGEX` 等判据锚点体系全部基于 Markdown 标题正则，扩展子标题与既有骨架同构，改动面最小。
- research 技术可行性对比表明确候选 B 的表格 schema"解析器需要更多新单测覆盖表格边界（空行/缺列），复杂度高于 A"——子标题方案回避此风险。
- 不锚定"代码块"是因为执行输出常含多行/特殊字符，用代码块包裹仅是排版建议，不宜作为机械判据的结构化锚点（正则难稳定锚定代码块起止）。

**留给 plan 阶段**：具体子标题文案措辞（中英文/是否加编号）由 plan 决定，不影响 spec 层判断。

### C2. FR-014 退出码语义核验的 MVP 边界

**结论**：MVP 仅做"退出码符号级"核验（0 = 命令成功/绿，非 0 = 命令失败/红），不做"输出内容是否语义匹配 issue 描述症状"的深层核验。

**依据**：
- spec 正文已自带"能力边界声明"章节（§128-138），明确声明本门"不能核验复现命令在语义上是否正确复现症状"，且这是 MAY 级功能（FR-014 标为 SHOULD，且自带"若超出能力边界 MUST 诚实声明"的退让条款）。
- research 风险清单 #1 的缓解策略聚焦"交叉核验命令是否真被执行"（存在性/执行痕迹层面），未要求语义匹配。
- 此结论不影响范围/安全，纯粹是"最小可行判据"选择，且 spec 已给出退让路径（能力边界声明），故可自动采纳为默认最小实现，无需用户拍板。

## GATE_DESIGN 待决问题（需用户裁决）

### Q1. FR-004 交叉核验的"对应关系"该判多严——严格到什么程度算匹配？

**为什么机械无法自决**：这是本 feature 的抗绕过核心（风险#1），也是"会不会误伤诚实用户"的核心（风险#2）。两端各有明确代价，且判定越严格实现和测试成本越高，不是能从既有代码或调研直接读出唯一答案的技术细节，而是一次"抓坏人 vs 放好人"的产品权衡，需要用户拍板要把门禁调多紧。

**通俗解释**：证据门要检查"你说执行过这条命令"和"你实际有没有真的跑过这条命令"是不是对得上。判断"对不对得上"可以松可以紧：
- 松判法：只要转录里出现过一次任意 Bash 命令执行记录，就算数，不细究是不是同一条命令。
- 紧判法：转录里的命令原文必须和 fix-report 里写的命令逐字精确相同才算数。
- 折中判法：允许命令有细微差异（比如多个空格、引号写法不同），但核心内容要能对上。

选太松：模型可以随便跑一条无关命令（比如 `echo ok`）就骗过门禁，等于没设防；选太紧：用户手打命令时哪怕换行/加了参数解释，明明是诚实执行了却被误判为不合规，体验很差、容易被投诉"机制太死板"。

| 选项 | 描述 | 影响 |
|------|------|------|
| A（推荐） | 折中匹配：归一化后（去空白/引号差异）做子串或核心片段比对 | 抗绕过与误伤风险相对平衡；实现和测试复杂度中等；仍存在被"部分相关但无效命令"擦边绕过的残余风险 |
| B | 松匹配：仅核验"transcript 中存在任意 Bash 执行记录"，不比对具体命令内容 | 实现最简单、几乎不会误伤，但抗绕过能力弱，容易被无关命令蒙混过关，可能重演 V008 病灶 |
| C | 严格匹配：命令原文逐字符一致才算通过 | 抗绕过能力最强，但对用户命令书写方式的容错为零，真实场景中大概率造成较多误伤（合法用户被无理由拦下） |

### Q2. EC-003"无法构造复现"场景的合法收口判据——要不要专门开一条例外通道？

**为什么机械无法自决**：spec 正文本身已明确标注这条"由 GATE_DESIGN 在 A vs A+B 之间拍板"（EC-003 原文），说明设计者已判定这是需要产品侧权衡的分支，不是技术细节。它决定了证据门是否会在"环境缺依赖、真做不出复现"的合理场景下把用户逼入死角。

**通俗解释**：有些问题因为当前环境跑不起来某个外部服务或运行时，用户确实没办法做出一个可执行的复现命令来证明"这问题是不是真的存在"。这时候要不要给一条"说明理由 + 用替代证据（比如指出历史修复的 commit 和对应代码现状）"的补充通道，让这种场景也能合法收口，而不是被证据门死死卡住？

不开通道：所有 no-op 一律要求可执行复现证据，逻辑最简单最一致，但会让"环境限制导致做不出复现"的合理用户被误伤，体验差、可能诱导用户伪造一个无意义的假复现来过关（反而制造新的绕过路径）。

开通道：给这类场景一个例外出口，体验更友好，但例外通道本身可能成为新的绕过缺口——如果"我这个环境跑不出来"这句话本身不需要证据就能被接受，模型完全可以滥用它逃避真做复现的要求。

| 选项 | 描述 | 影响 |
|------|------|------|
| A（候选 A 单选） | 不开例外通道，无法构造复现的场景一律要求补充结构化替代证据（如具体 commit hash + 代码现状摘录）才能收口，同样要过结构化字段校验 | 一致性最强、抗绕过风险最低；但对真实环境受限场景仍有一定摩擦，需要用户额外说明和举证 |
| B（A+B 组合，spec 已标注为降级选项） | 开例外通道：允许"无法构造复现"作为一个显式声明的收口路径，配合候选 B 的对账骨架做替代举证 | 对合理场景更友好；但需要额外设计"如何证明确实无法构造复现"本身的证据要求，否则例外通道会被滥用为新的绕过口子，设计与测试成本更高 |
