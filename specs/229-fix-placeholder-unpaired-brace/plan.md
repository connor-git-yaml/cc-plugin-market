# 修复规划 — F229 占位符「不成对花括号」绕过收口

对应 `fix-report.md` 已确认的 Root Cause：两条花括号占位判据（`PLACEHOLDER_BRACE_REGEX` / `CANONICAL_PLACEHOLDER_REGEX`）
都把"闭合"错误当作判据的必要组成部分，而闭合与否与"是否仍是未替换的模板占位符"无关——删掉一个 `}` 即可

锚点收窄为"存在未替换的模板起始标记 `{`"，不新增第三条判据、不改变四段 OR 结构。

## 2. 为何两条正则的收口方式不同

**共同前提**：两条正则要解决的都不是"这段花括号是不是占位符"的内容猜测，而是复用 F228 已确立的
结构性边界——文本处于"代码区"还是"散文区"。真正的差异只在于**它们各自扫描的是哪一份文本**。

| | `PLACEHOLDER_OPEN_BRACE_REGEX`（原 `PLACEHOLDER_BRACE_REGEX`） | `CANONICAL_PLACEHOLDER_REGEX` |
|---|---|---|
| 扫描对象 | **两个消费点，输入不同**（本表原先只写了第一个，是事实错误）：<br>① 四段 OR 第 **3** 段 — `placeholderScanText = stripCodeRegions(proseBody)`，**已剥离**闭合 fenced 块 + 行内 code span；<br>② 四段 OR 第 **4** 段 — 直接吃**未剥离**的 `proseBody`，与 `strippedChars <= MIN_SECTION_BODY_CHARS` 取合取（F228 R3-1 的代码区豁免边界判据）。故"该常量只作用于已剥离文本"这一说法**不成立**，第 4 段刻意在原文上找花括号，再用剥离后剩余散文量作判别锚点 | `proseBody`（**未剥离**，四段 OR 第 2 段直接吃原文） |
| 为何未剥离/已剥离不同 | 通用花括号判据服务"作者在实质散文之外引用真实代码字面量"这一豁免场景（如 `` `{path: null, ambiguous: true}` ``），真实代码的花括号理应被排除，所以让它在剥离后的文本上运作 | canonical 判据专门堵"模板占位符包一层反引号也不能免检"这条 F228 R2-2 CRITICAL-1 修复路径——如果也在剥离后文本上跑，`` `{根本原因一句话总结}` `` 会被行内 code span 剥离掉而漏检，因此必须锚定在剥离**之前**的原文 |
| 剩余的"真实代码花括号"如何排除 | 第 3 段消费点已被 `stripCodeRegions` 结构性清空——不需要正则自己再做任何语义判别，故可以退化为最朴素的 `/\{/`：只要剥完代码区后还剩一个 `{`，就一定不是"作者引用的代码"。第 4 段消费点则不靠正则排除，靠 `strippedChars` 阈值排除 | 无法借助剥离，只能在同一份原文里正则自己判别"这是代码/JSON 字面量" vs "这是中文占位符"——继续依赖两项既有判别力：① `(?=[^}]*[一-鿿])` 前置要求就近出现 CJK 表意文字；② `[^}:]*` 排除 ASCII 冒号（协调方复审结论：ASCII 冒号才是对象字面量/JSON 键值结构的可靠标志，ASCII 引号不可靠，故不进排除集） |
| 本次需要动什么 | 直接删掉整个"必须闭合"的形态约束（`[^}]*\}` → 空，只留起始标记 `\{`）。**措辞订正**：`/\{/` 命中的是**任何 ASCII U+007B，不区分 Markdown 转义**——`\{`（反斜杠转义的花括号）同样命中，该正则并不检查转义，故不应表述为"未转义的模板起始标记" | 把"闭合边界"从"必须是 `}`"放宽为"`}` 或**行尾**"，且拆成两个 alternation 分支（成对分支保留旧形态、不成对分支排除 `\n` 并配 `/m`）；判别力两项（CJK 前置 + 排除 ASCII 冒号）在两分支中逐字保留、不能删 |

**用具体例子验证"为何只能放宽边界、不能整体退化"**：以 `{"claim":"症状已消除","command":"npx vitest run"...}`
（含 CJK「症状已消除」、也含 ASCII 冒号）为例——若 canonical 判据也退化为 `/\{/`，这条真实 JSON 证据会被误判为
占位符（假阳性，破坏既有合规 no-op 复现对账写法，F216 C1 已依赖此判据不误伤）。保留 `[^}:]*` 的冒号排除后，
即便新增不成对分支，正则在遇到 `:` 时仍会在**该起点**匹配失败（`[^}:]*` 的贪婪扩张一旦遇到冒号即被迫停止，
而停止点既非 `}` 也非行尾，该起点的匹配尝试失败）。

**这一豁免的作用域必须收窄陈述（Codex 核实的事实错误订正）**：ASCII 冒号排除**只是每个 `{` 起点的局部条件**，
不是"整段文本含冒号即整体豁免"的全局性质。正则引擎在第一个 `{` 失败后会继续从**后续每一个 `{`** 重新起匹配，
因此同一段 JSON 里只要存在某个 `{`，其后到 `}`/行尾之间恰好无 ASCII 冒号却有 CJK，新判据依然会命中。
反例（实测命中）：`{"claim":"{症状已消除` —— 第一个 `{` 因随后的 `:` 失败，但第二个 `{` 之后是纯 CJK 直到行尾，
不成对分支命中。故本文档与代码注释一律只能说"该 `{` 起点因 ASCII 冒号被豁免"，不得泛化为
"JSON 含冒号即可可靠豁免"。真正可靠的 JSON 豁免手段仍是 F216 的 `stripReconSubblock` 结构性剔除与 F228 的代码区剥离。

## 3. 回归风险评估

结构性判据放宽闭合要求后，唯一可能引入误报（合法制品被误判为占位）的路径是"散文中出现真实的、
未被 code span/fence 包裹的裸 `{` 字符，且不是模板占位符"。逐条评估：

| 可能误报的合法制品形态 | 是否构成真实风险 | 说明 / 需要的测试 |
|------------------------|------------------|--------------------|
| 作者在**行内 code span 或 fenced 块内**书写真实代码字面量（如 `` `{path: null, ambiguous: true}` ``、fenced JSON） | **不构成风险** | `PLACEHOLDER_OPEN_BRACE_REGEX` 只扫描 `stripCodeRegions` 之后的文本，闭合与否代码区都已被结构性清空；既有回归测试（L1986 A、L1998 B、L2291/2303/2321 三条"R3 回归保护"）全部沿用，逐字复用即为验证 |
| 作者在**散文正文**直接书写真实对象字面量，且**完整闭合**（如裸写不加反引号的 `{ ok: true }`） | **不构成新增风险** | 旧正则本就要求闭合、已经会命中判占位（因为旧正则本身不区分"代码"与"占位符"，纯散文里裸写闭合花括号在旧实现下也判占位）；本次改动不改变"闭合花括号在纯散文里判占位"这一既有行为，风险面不变 |
| 作者在**散文正文**提及花括号语法本身但**未闭合**且**该章节剩余散文里没有 ASCII 冒号**（如讨论 Handlebars/正则 `{n,m}` 语法但手误漏打右括号，随后是大段中文说明） | **新增边界，需显式钉住** | 这是本次修复**唯一** 新扩大的判定面：旧正则因"要求闭合"而对任何未闭合花括号一律不命中，新正则（尤其 canonical 分支新增的 `$` 出口）会命中。但这恰是 Root Cause 陈述的直接推论——"占位符的语义标志是存在未替换的起始标记，与闭合无关"对任何未闭合花括号一视同仁地生效，不是本次改动的意外副作用，而是判据设计的题中之义。缓解方式：作者提及花括号语法本身时应完整闭合或改用全角括号描述，不构成代码缺陷；本计划要求在 §4 补一条 characterization 测试把这一权衡显式钉住，供未来复审 |
| `PLACEHOLDER_OPEN_BRACE_REGEX` 第 4 段 OR（`strippedChars<=20`）随常量收口自动收紧 | **不构成新增风险，但需回归验证** | 该分支要求"原文含花括号 **且** 剥离代码区后散文不足阈值"双重条件；`strippedChars<=20` 这道阈值门槛保持不变，只是"原文含花括号"的判定从"必须闭合"放宽为"存在裸 `{`"——语义仍是"整段正文被代码区形态包裹、剥完所剩无几"，既有 §4 表中 E/R3 三条"散文空洞但 fenced code 撑长度"回归测试足以覆盖 |
| `stripCodeRegions` / `stripInlineCodeSpans` / `computeFenceRegions` 本身的行为 | **不受影响** | 本次未改动这三个函数，跨行 code span、4 空格缩进块两条既有 characterization（L2206/L2218）继续保持"当前行为"不变 |
| **闭合** code span / fenced 块内的**未闭合代码花括号**，且**同一行**该 `{` 之后还有中文散文（如 `` `{` `` 之后紧跟中文说明，或围栏内 `if (x) { // 这里返回默认值`） | **新增边界，非缺陷，但必须写明** | canonical 判据按 F228 R2-2 的刻意设计**不吃代码区豁免**（它扫的是未剥离的 `proseBody`），故这类形态会被不成对分支命中。这不是本次新引入的取舍，而是 F228 既定取舍的延伸——今天 `` `{中文}` `` 写在 code span 里本就命中；F229 只是把"成对"扩到"不成对"。**因此不能继续声称"唯一新增面是散文中的裸花括号"**：代码区内的不成对花括号只要同行后随 CJK，同样进入新增面。缓解方式与 F228 一致：真实代码若含中文注释且花括号不成对，请写成成对形态或把中文移出该行 |
| 未闭合围栏 / 跨行代码片段（围栏内 `{` 在**行尾**、CJK 在**下一行**） | **不构成风险（R1 收口后）** | 不成对分支排除 `\n` 且 `$` 为行尾，`{` 在行尾时同行无 CJK → lookahead 失败；成对分支需要真实 `}` → 失败。这正是 R1 回归修复的核心场景，已由测试 `R1-a` 钉住 |

**结论**：需新增测试锁定的风险面有两条——① "未闭合裸花括号 + 该 `{` 起点之后无 ASCII 冒号"（散文形态）；
② 代码区内不成对花括号同行后随 CJK（F228 既定取舍的延伸）。其余路径要么被既有回归测试覆盖、要么行为不变。

### 4.3b `PLACEHOLDER_OPEN_BRACE_REGEX` 真隔离用例（R2 收口）

- **输入**：`**Root Cause**: 具体成因待补充，模板字段 {reason: 尚未替换为真实内容，此处补足足够长度的中文说明以越过最小正文阈值。`
