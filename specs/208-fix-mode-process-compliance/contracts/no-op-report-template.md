# 合同："确认无需改动"精简报告模板与机械判据锚点（FR-003/FR-004/FR-012a）

**修改文件**：`plugins/spec-driver/skills/spec-driver-fix/SKILL.md`（Phase 1 内新增判定分支）

## 模板（写入 `{feature_dir}/fix-report.md`）

```markdown
# 问题核实报告（无需改动）

## 问题描述
{用户原始描述}

## 判定依据
{为何判断问题已不存在/无需代码改动的具体证据：如指向已生效的历史修复 commit、
实际复现测试结果、相关代码路径现状摘录等——不得是空泛的"经检查确认无问题"}

## 交叉核实委派
{委派的子代理角色 + 核实结论摘要}
```

## 机械判据锚点（`checkArtifactSection` 消费此契约）

| 判据 | 规则 |
|------|------|
| 制品存在性 | `{feature_dir}/fix-report.md` 存在 |
| 非空 | 文件去空白后长度 > 0 |
| 必填章节存在 | 正则 `/^##\s*判定依据\s*$/m` 命中 |
| 非占位空壳（F228 四段判据） | 正文（去空白）长度 > 20 字符；**且** canonical 中文模板占位符（形如 `{根本原因一句话总结}`，含中日韩表意文字、不含 ASCII 冒号）在正文原文上不命中——该判据**跨代码区不豁免**；**且**通用花括号占位符（正则 `/\{[^}]*\}/`）在剥离**闭合**围栏与行内 code span 之后的文本上不命中——未闭合围栏（开围栏行直到文件尾都未闭合）不剥离，转义反引号 `` \` `` 不作 code span 定界符；**且**若正文原文含花括号，则剥离代码区后的实质散文（去空白）须 > 20 字符——代码区豁免只服务于"作者在实质散文之外引用代码"，不服务于"整段正文就是包在代码里的占位符" |

**标题字符串为 canonical、唯一匹配依据**——SKILL.md 中生成该模板时必须逐字使用 `## 判定依据`，不得意译或改写为近义标题（如"判断依据"“核实依据”），否则会被判定器误判为"缺失必填章节"（FR-012 要求机械可判，不做语义近似匹配，属于 FR-009 可选范畴之外）。

## no-op 交叉核实委派判据（FR-004/FR-012b，codex plan 审查 W-2 处置）

no-op 收口的最低委派要求 = ≥1 条 **no-op 核实类**委派：对 `subagent_type` 与 `description` 级联匹配 `/verify|spec-review|quality-review|review|验证|审查|核实|确认/i`。SKILL no-op 分支的 canonical 委派调用文本为 `Task(description: "交叉核实无需改动判定", ...)`（desc 含"核实"确保命中）。

**已明示的残余风险（FR-009 可选范畴，本期接受）**：机械判据只能验证"发生过一次核实语义的真实委派"，无法验证委派内容与 no-op 判断结论的深层相关性（如委派核实的是无关代码路径）。深度语义相关性校验属 FR-009 深度语义识别，不在本期机械底线内；缓解依赖 `## 判定依据` 正文的非空+非占位下限与评测侧人工抽查。

## 与"修复收口"形态的判据区分（同一 `checkArtifactSection` 函数复用，仅 `requiredHeading` 参数不同）

| 收口形态 | 制品路径 | 必填章节标题正则 |
|---------|---------|-----------------|
| 修复收口 | `{feature_dir}/fix-report.md` | `/Root Cause/i`（既有 Phase 1 模板已含 `**Root Cause**:` 行，不改动既有模板文案） |
| 修复收口 | `{feature_dir}/verification/verification-report.md` | 无必填章节要求（仅存在性 + 非空） |
| no-op 收口 | `{feature_dir}/fix-report.md`（no-op 变体） | `/^##\s*判定依据\s*$/m`（本契约新增） |

**判定器如何区分"这是修复收口的 fix-report.md 还是 no-op 收口的 fix-report.md"**：文件路径相同（均为 `{feature_dir}/fix-report.md`），区分依据是**内容特征**而非路径——若文件内容命中 `## 判定依据` 标题，判定为 no-op 变体走 no-op 判据集；否则按修复收口判据集（`Root Cause` 章节 + 委派角色匹配）校验。两种模板的标题集合刻意设计为互斥（修复收口模板不含 `## 判定依据` 标题，no-op 模板不含 `Root Cause` 字样），避免误判。

**双锚点取严规则（codex implement 审查 W-1 处置）**：若 fix-report.md 同时命中 `Root Cause` 与 `## 判定依据` 两个锚点（模板被混写），closureForm 判为 **repair**（取更严的修复收口判据集）——既声称完成根因分析就必须满足 verification-report + implement/verify 双角色配额，不允许借 no-op 的更低门槛绕过。
