## User Scenarios & Testing *(mandatory)*

### User Story 1 - 无改动收口场景被正确拦截并引导补救（Priority: P1）

一个 fix 会话中，agent 判断"问题已修复/无需改动"，但实际上既未建立特性目录、也未生成 fix-report.md、也没有对判断结论做任何子代理委派核实，就试图在最终消息里直接给出一份"已完成"的收口陈述。系统必须在该会话即将结束时拦截这次收口，并向 agent 反馈缺失了什么、该如何补救（走完整流程或走轻量的"确认无需改动"出口）。

**Why this priority**: 这是本次立项要解决的核心缺陷（F206 证据显示的地方病，20%-29% run 受影响）。没有这条能力，其余功能都没有意义。本 Story 全部验收场景默认 `enforcement=block`（默认档，见 FR-015 行为矩阵）。

**Independent Test**: 可独立测试——构造一个"0 委派、无制品、直接输出完成陈述"的 fix 会话（可用 mock transcript 或真实 headless 调用复现），验证会话结束时被结构化机制拦截而非静默放行。

**Acceptance Scenarios**:

1. **Given** 一个 fix 会话中 agent 未建立特性目录、未产出 fix-report.md、transcript 中无任何子代理委派记录，**When** agent 尝试结束会话并给出"已完成"式收口陈述，**Then** 系统必须阻断本次收口，并向 agent 返回具体、可执行的补救指引（例如"请通过委派完成诊断并生成 fix-report.md，或走"确认无需改动"出口并完成最低限度核实"）。
2. **Given** 拦截发生后，agent 按照反馈指引补齐了必需制品与至少一次委派核实，**When** agent 再次尝试结束会话，**Then** 系统必须放行本次收口。
3. **Given** 一个 fix 会话完整走完了标准流程（建立特性目录、产出 fix-report.md、经过子代理委派完成诊断与修复），**When** agent 结束会话，**Then** 系统不得产生任何拦截或额外提示（不影响正常流程）。

---

### User Story 2 - "确认无需改动"成为一等公民收口路径（Priority: P1）

用户报告的问题经排查后确实不存在（例如已被之前的改动修复、或用户描述有误），agent 需要有一条**显式、合规**的轻量路径来收口这类诚实场景，而不是被强制清单逼着要么伪造制品要么被反复拒绝。

**Why this priority**: 与 Story 1 是同一枚硬币的两面——没有合法出口，强制清单会误伤真实的"无需改动"场景，逼出 reward hacking（伪造委派/空转制品）。技术调研明确将此列为必要设计（方案 d），且与 Story 1（方案 a/c）互为前提，共同构成 P1。

**Independent Test**: 构造一个"用户报告的问题经核实确实已被修复/不存在"的 fix 会话，验证 agent 能通过该轻量路径合规收口（产出精简版诊断记录 + 完成最低限度的委派核实），且不被 Story 1 的拦截机制误判为坍塌。

**Acceptance Scenarios**:

1. **Given** agent 排查后判断问题确实无需代码改动，**When** agent 走"确认无需改动"路径，产出精简版判定记录并完成至少一次委派核实该判断，**Then** 系统应认可这是一次合规收口，不予阻断。
2. **Given** agent 判断"无需改动"但未做任何委派核实（0 委派），**When** agent 尝试通过该轻量路径收口，**Then** 系统仍应判定为不合规并阻断（轻量路径不是零验证的免检通道）。

---

### User Story 3 - 依从性判据来自结构化状态而非模型自陈（Priority: P2）

无论走完整路径还是轻量路径，判断一次 fix 会话是否"合规收口"的依据必须来自客观可验证的结构化信号（制品文件是否存在、委派记录是否真实发生），而不是模型在输出文本中"自称已完成"。

**Why this priority**: 这是 Story 1/2 得以成立的底层能力，但从用户可观测的角度看是内部机制而非独立可体验的功能，故列为 P2（支撑性而非独立交付价值）。

**Independent Test**: 分别构造"模型自称已委派但 transcript 无对应委派记录"与"模型自称未完成但 transcript 有完整委派记录"两个反例场景，验证判定结果与 transcript 客观记录一致，而非与模型自陈文本一致。

**Acceptance Scenarios**:

1. **Given** 一个会话的最终陈述文本中声称"已完成 3 次委派诊断"，但 transcript 中实际委派记录为 0 次，**When** 依从性判定执行，**Then** 判定结果必须以 transcript 记录为准（判定为不合规），不采信模型自陈。
2. **Given** 一个会话的委派记录、制品文件均满足要求，**When** 依从性判定执行，**Then** 判定结果必须为合规，无论最终陈述文本表述是否完整。

---

### User Story 4 - 阻断机制有界化，不产生死循环（Priority: P2）

拦截机制在同一会话内被触发后，如果 agent 反复未能补齐要求（例如多次误判或环境异常），系统不能无限期阻塞该会话，必须在达到上限后以可追溯的降级方式放行。

**Why this priority**: 防御性质的能力，保障 Story 1 机制本身不会引入新的可用性风险，重要但依附于 Story 1 存在，故 P2。本 Story（阻断计数与降级放行）仅适用于 `enforcement=block` 档（`warn` 不阻断故无计数，见 FR-015 行为矩阵）。

**Independent Test**: 构造一个持续触发拦截条件的会话（agent 始终未能补齐要求），验证系统在达到预设阻断次数上限后自动放行，并在放行结果中留下可识别的降级标注。

**Acceptance Scenarios**:

1. **Given** 一个会话已被拦截达到预设次数上限（2 次），**When** agent 再次尝试收口且仍未满足合规判据，**Then** 系统必须放行本次收口，同时在记录中标注本次为降级放行（而非静默判定为合规）。
2. **Given** 一个会话尚未达到阻断次数上限，**When** agent 未满足合规判据尝试收口，**Then** 系统必须继续阻断（而非提前放行）。

---

### User Story 5 - 非 fix 会话不受影响（Priority: P2）

同一套插件机制会挂载到所有会话（feature/story 模式、普通交互式会话、评测 headless 会话），新增的拦截逻辑必须只对 fix 会话生效，不能误伤其他场景。

**Why this priority**: 直接影响机制能否安全上线，是 Story 1 的必要边界条件，但本身不构成独立用户价值，列为 P2。

**Independent Test**: 分别在 feature 模式会话、story 模式会话、普通问答会话中触发结束流程，验证均不产生任何 fix 依从性相关的拦截或提示。

**Acceptance Scenarios**:

1. **Given** 一个 feature 模式会话正常结束，**When** 会话尝试收口，**Then** 不得触发 fix 模式依从性拦截。
2. **Given** 一个未展开任何 spec-driver 技能的普通问答会话结束，**When** 会话尝试收口，**Then** 不得触发 fix 模式依从性拦截。

---

### Edge Cases

- **正常修复流程不受影响（关联 Story 1 场景 3、FR-001/FR-002）**：完整走完 fix 流程（建目录、写 fix-report.md、经委派诊断与修复）的会话，收口时不应有任何额外阻断或可感知的延迟。
- **诚实 no-op 不被误拒（关联 Story 2、FR-003/FR-004）**：用户确实报告了一个已经不存在的问题，agent 走轻量出口且完成最低限度核实时，不能被判定为坍塌。
- **Stop 阻断死循环的有界化（关联 Story 4、FR-006）**：阻断次数上限为 2 次；达到上限后必须放行并标注降级状态（`[GATE-DEGRADED]`），不能无限阻塞会话。
- **非 fix 会话误伤（关联 Story 5、FR-007）**：会话自判信号必须取自模型无法遗弃的动作——用户调用 fix 技能时由 harness 展开注入 transcript 的痕迹，而非任何住在 SKILL 流程内的标记文件或模型自愿产生的状态：坍塌 run 的本质特征就是可以整体遗弃 SKILL 中的任何"固定步骤"（F206 证据：连 init-project.sh 都不跑），若以流程内标记为门控信号，"无标记"将无法区分"坍塌的 fix 会话"与"非 fix 会话"，导致要么全量误伤要么全量漏拦。基于技能展开痕迹判定则三态清晰：有展开且达标→放行；有展开且不达标→阻断；无展开→非 fix 会话，零接触放行。
- **headless 与交互式双场景生效（关联 FR-001、FR-008）**：评测环境（`claude --print` 加 `--plugin-dir` 指向本仓库源码、无 `--bare`）与用户交互式会话都必须被机制覆盖，不能只在一种场景下生效。
- **reward hacking / 规避形态防御（关联 FR-005、FR-009）**：判据必须能识别并拒绝"伪造委派记录"或"制品文件占位但内容空洞不具实质"等已知规避形态，把这些列为清单校验的反例，而非只做"文件是否存在"的表层检查；判据来源必须是 harness 层写入的客观记录（如 transcript 中的委派工具调用记录），而非可被模型自由生成的文本。
- **轻量验证路径与完整路径并存（关联 FR-002）**：本仓库已有的"轻量验证路径"（小修复场景 4a/4b 并入 4c）不构成豁免，判据清单在两条路径下复用同一核心校验逻辑，仅完整路径额外校验更多阶段性制品。
- **非展开触发形态不产生展开痕迹（关联 Story 5、FR-007/FR-008）**：通过 stdin 纯文本传入的请求，或未经 CLI 展开、仅在文本中提及技能名的会话，均不会在 transcript 中留下 harness 注入的展开痕迹，必须按非 fix 会话处理（零接触放行）；交互式 slash 展开与 headless 位置参数展开两种合法展开形态的痕迹识别均须覆盖，不能只识别其中一种。
- **同一会话内多次/多技能展开（关联 FR-007）**：会话中途从 feature 展开到 fix，或同一会话内多次展开 fix 技能时，判定必须锚定"当前收口所对应的最新一次 fix 技能展开"，仅统计该次展开之后产生的制品与委派记录，不得把更早的、属于其他展开或已收口周期的记录计入本次判定。
- **并发 fix 会话不共享阻断状态（关联 Story 4、FR-006）**：多个 fix 会话并发运行时，各自的阻断计数与降级状态必须以稳定的会话级标识（与各自 transcript 一一对应）隔离存储，不得相互覆盖或共享计数导致一个会话的阻断历史影响另一个会话的判定。
- **transcript 异常与 hook 内部异常的失败开放语义（关联 FR-013）**：transcript 缺失、不可读、格式不识别、体积过大或解析超时，以及 hook 自身抛出未预期异常时，hook 必须捕获全部错误并按非 fix 会话放行，同时把 degraded 原因（如 `transcript-unavailable`）写入结构化落盘记录，供事后审计发现潜在漏拦，禁止让脚本崩溃行为隐式决定阻断或放行。
- **与既有非阻断型 Stop hook 并存（关联 FR-010）**：仓库已有的非阻断型 Stop hook（任务提醒类）与本次新增的阻断型 fix 依从性 hook 同时挂载时，既有 hook 的行为与输出不受影响，且两者的输出内容可通过稳定前缀相互区分，不产生混淆或覆盖。
- **配置降级不静默（关联 FR-015）**：项目显式配置 `warn` 时，不合规收口不被阻断，但判定结论必须以稳定前缀反馈并按 `block` 同口径落盘；配置 `off` 时 hook 零接触快速退出（`off` 的审计痕迹是项目配置文件本身及其版本历史，不额外落盘）；配置缺失或取值非法时按默认 `block` 执行并附 `config-degraded` 诊断，不得把解析失败静默降级为 `off`（否则一个损坏的配置文件即可整体关闭保障机制），也不得让配置解析失败落入 FR-013 的 fail-open 分支（两类回落的边界与顺序见 FR-015 判定顺序）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供一个**住在模型上下文之外**的强制拦截点，在 fix 会话试图结束（收口）时，依据客观可验证的判据检查本次会话是否达到最低合规要求；不达标时必须阻断本次收口。本条及 US1/US4 的"阻断"语义仅适用于 `enforcement=block`（默认档，含未配置场景）；`warn`/`off` 档的对应行为由 FR-015 行为矩阵定义。**[必须]**——这是解决"读了 SKILL 但选择遗弃流程"问题的唯一结构性手段，去掉后核心需求（防仪式坍塌）无法实现。
- **FR-002**: 判定"是否达到最低合规要求"的判据清单必须按收口形态区分三支合法/非法组合：(i) **修复收口**（含轻量验证与完整验证两种 Phase 4 变体）——最低必需为 fix-report.md + verification/verification-report.md + ≥2 次真实子代理委派（其中须含 implement 类与 verify 类各 ≥1）；(ii) **no-op 收口**（"确认无需改动"出口）——最低必需为 fix-report.md 的 no-op 判定记录（含判定依据）+ ≥1 次用于交叉核实该判断的委派；(iii) 任何形态下"零委派 + 无任何诊断制品"均一律判定为不合规，不存在豁免。plan.md/tasks.md 等阶段性制品的存在性校验属于 FR-009 可选加强项，不构成本条最低门槛。**[必须]**——去掉按形态分层的门槛，修复收口最低门槛过低（仅需 1 次装饰性委派）即可被 Goodhart 绕过并复刻 F206 的坍塌模式；去掉"零委派零制品不合规"底线则无法防坍塌本身。
- **FR-003**: 系统必须提供一条显式的"确认无需改动"收口路径，允许 agent 在排查后判断无需代码改动时，通过产出精简版判定记录并完成最低限度的委派核实来合规收口，而非被强制清单逼迫伪造制品或被反复拒绝。**[必须]**——去掉该路径，诚实的"无需改动"场景将被 FR-001/FR-002 的清单机制系统性误判为坍塌，损害合法场景。
- **FR-004**: "确认无需改动"路径的最低限度委派核实要求不得为零——该路径必须要求至少一次委派用于交叉核实"确实无需改动"这一判断，防止该出口被滥用为绕开 FR-001 的零验证捷径。**[必须]**——去掉这一底线，FR-003 会退化为坍塌行为的合法遮羞布，产生比现状更隐蔽的 reward hacking 风险。
- **FR-005**: 依从性判据的信号来源必须是 harness 层写入的、模型无法伪造或遗漏的客观记录（如会话 transcript 中的子代理委派调用记录、制品文件的存在性与实质内容），而非模型输出文本中的自陈声明。**[必须]**——去掉此约束，判据可被模型的自然语言输出直接操纵，机制形同虚设，与已证伪的三轮 prompt 级守卫同构。
- **FR-006**: 阻断机制必须有界：同一会话内因不合规被阻断的次数上限为 2 次；达到上限后（即第 3 次仍不满足合规判据时），系统也必须放行本次收口，并在放行结果中留下可追溯的降级标注——标注需同时写入 (1) Stop hook 返回的 `reason` 文本（含 `[GATE-DEGRADED]` 标记）与 (2) `record-workflow-run.mjs` 落盘的 workflow-run-summary JSONL 事件的对应字段，而非静默视为合规。阻断计数必须以稳定的会话级标识为键（与该会话的 transcript 一一对应存储），并发的多个 fix 会话不得共享同一份计数或降级状态。**[必须]**——去掉有界化设计，机制存在导致会话无限阻塞的可用性风险，损害产品可用性且降低用户对该机制的信任，最终可能被绕过（如强制中断会话）；去掉会话级计数隔离，并发场景下会出现互相污染判定的错误行为。
- **FR-007**: 拦截机制必须能够区分"当前会话是否为 fix 模式会话"，仅对 fix 会话生效；非 fix 会话（feature/story 模式会话、普通交互式会话）即便挂载了同一套插件机制，也不得被误判为需要合规检查。同一会话内可能存在多技能展开（如 feature 会话中途展开 fix）或同一技能的多次展开，判定必须锚定"当前收口所对应的最新一次 fix 技能展开"，仅统计该次展开之后产生的制品与委派记录，不得跨展开周期累计。**[必须]**——去掉此区分，所有会话都会被无差别拦截，产生大规模误伤，机制无法上线；去掉展开边界锚定，多技能/多次展开场景下判定窗口错位，可能把早前展开周期的记录误算入当前收口或反之。
- **FR-008**: 拦截机制必须在交互式会话与 headless（评测自动化）会话两种运行形态下均生效，且不依赖任何需要模型主动配合写入的中间状态标记（因为坍塌行为的特征之一正是不产生任何"流程自愿留下的痕迹"）。该约束同样适用于会话自判信号本身：自判必须基于 harness 写入的客观痕迹（如 transcript 中的 fix 技能展开记录），任何住在 SKILL 流程内的标记文件均不得作为门控信号（其缺失不得导致拦截被跳过）；交互式 slash 展开与 headless 位置参数展开两种合法展开形态的痕迹识别均须覆盖。**[必须]**——去掉 headless 覆盖，机制无法在评测环境中验证有效性，也无法解决 F206 证据所指出的评测污染问题；依赖模型主动标记则与坍塌行为的本质矛盾（坍塌就是不写标记）。
- **FR-009**: 判据清单在校验制品与委派记录时，应能识别更深层的语义级规避形态（如委派内容与问题描述的相关性分析、判断结论与制品内容的一致性核查），而非仅做浅层的"文件是否存在"检查。其中机械可判的规避识别子集（制品非空且含模板必填章节、委派角色类型与收口形态匹配）已由 FR-012 升级为必须项；本条聚焦于超出机械可判范畴的深度语义级识别。**[可选]**——不做深度语义级校验时核心拦截能力与机械可判底线（FR-012）仍然成立，能挡住"完全零委派零制品"及"占位空壳"两类坍塌，但会降低机制对更隐蔽、需要语义理解才能识别的规避行为的防御深度，是可维护性与鲁棒性层面的加强项，非本次范围的准入门槛。
- **FR-010**: 拦截机制阻断收口时，向 agent 返回的反馈内容必须具体、可执行，明确指出缺失了什么（例如缺少 fix-report.md、缺少委派记录）以及应走哪条合法路径补救（完整路径或"确认无需改动"路径）；反馈文本必须带稳定可识别的前缀，以便与仓库既有非阻断型 Stop hook（如任务提醒）的输出相区分，也便于评测口径统计。**[必须]**——去掉具体反馈，agent 面对模糊拒绝无法有效补救，会退化为反复试错甚至放弃，实测（harness-verification.md 实锤 3）已证明反馈内容质量直接驱动模型的正确补救行为；去掉稳定前缀，两类 Stop hook 的输出在评测日志与用户可见文本中难以区分，影响可观测性与统计口径。
- **FR-011**: 判据清单不得以任务标识符、任务描述文本内容作为判定依据；所有判定均只能基于制品存在性、委派记录、会话内客观状态。**[必须]**——去掉此约束，方案会与 F206 已明确的红线冲突（"不得特判任务 ID"），导致方案在评测场景下过拟合失效并被判定为无效交付。此约束与本节末尾"约束与非功能需求"中的判定路径边界（C-001~C-003）共同构成本方案的红线集合。
- **FR-012**: 判据清单必须包含一套机械可判的实质性底线校验，对制品与委派记录做结构化核验而非仅存在性检查：(a) 制品文件非空且含其对应模板的必填章节（如 fix-report.md 须含根因分析/判定结论对应小节，no-op 判定记录须含判定依据小节）；(b) 委派记录的角色类型必须与收口形态匹配（修复收口须能在委派记录中识别出至少 1 次 implement 类与 1 次 verify 类委派，no-op 收口须能识别出至少 1 次用于交叉核实的委派）。该校验必须纯本地机械可判——仅基于文件系统读取与 transcript 结构化字段解析完成，不得引入任何 LLM 调用或额外子代理委派。**[必须]**——去掉此项，占位空壳制品（如仅含标题的空文件）与角色不匹配的装饰性委派（如仅委派一次与问题无关的调研）可以骗过 FR-002 的最低门槛计数，复刻 F206 R2 揭示的"合规绕过"教训，使整套结构化保障退化为可被凑数糊弄的表面合规。
- **FR-013**: hook 必须捕获全部内部异常与输入异常场景，不允许脚本崩溃隐式决定阻断或放行：当 transcript 缺失、不可读、格式不可识别、体积过大或解析超时，或 hook 自身运行时抛出未预期异常时，一律按非 fix 会话放行（宁可漏拦一次不可全局误伤或阻塞全部会话），但必须把本次判定退化为 degraded 状态的结构化诊断信息（含具体原因，如 `transcript-unavailable`、`parse-timeout`、`internal-error`）写入落盘记录，使这类"漏拦"在事后审计中可被发现而非彻底隐没。**[必须]**——去掉此约束，hook 的异常行为未定义，可能导致脚本崩溃后被平台判定为阻断从而误伤一切会话（包括 feature/story 等非 fix 场景），或者静默失效导致所有异常场景无声放行且不可追溯，两种后果都会破坏机制的可用性或可审计性底线。
- **FR-014**: 工作流运行记录组件（`record-workflow-run.mjs`）从"只记录"升级为"可校验"时必须保持向后兼容：默认状态（未显式启用强校验参数时）保持既有只记录语义不变；现有全部调用方（fix/story/implement/doc/resume 五个 SKILL 的收口调用）在不修改自身调用代码的情况下行为不变；仅 fix 模式显式启用本次新增的强校验逻辑。**[必须]**——去掉向后兼容保障，该组件的升级会直接破坏其余四个模式（story/implement/doc/resume）既有的收口链路，属于跨模式回归，代价远超本次仅针对 fix 模式的收益范围。
- **FR-015**: 依从性拦截的强制程度必须提供项目级配置（`spec-driver.config.yaml`），供通用用户按自身场景显式调整，默认值必须为 `block`。三档行为矩阵（本 spec 中 FR-001/US1/US4/FR-006 的阻断语义均仅适用于 `block` 档）：

  | 维度 | `block`（默认） | `warn` | `off` |
  |------|----------------|--------|-------|
  | 不合规收口 | 阻断并反馈补救指引（FR-001/FR-010） | 放行，但以稳定前缀反馈缺口 + 与 `block` 同口径落盘（含缺失项清单） | 零接触放行，不判定不输出 |
  | 阻断计数/上限 | 同一会话至多阻断 2 次，第 3 次降级放行（FR-006） | 不适用（不阻断故无计数） | 不适用 |
  | 判定过程异常 | fail-open 放行 + degraded 诊断落盘（FR-013） | 同 `block`（FR-013 同样适用） | 不适用（不进入判定） |
  | SC-001/SC-002 统计口径 | 计入 | 计入（loud 记录横向可比） | **排除**——`off` 是显式 opt-out，其选择本身记录在项目配置文件中（用户仓库的配置与其版本历史即审计痕迹），不产生落盘记录以维持零接触语义 |

  **判定顺序（类型化回落边界，实现必须遵守）**：(1) 首先以**非抛出式**解析读取配置得出 `enforcement`——配置缺失、不可读或取值非法时统一得出 `enforcement=block` 并附带 `config-degraded` 诊断（回落到默认强度而非静默关闭）；(2) `enforcement=off` → 立即零接触退出；(3) 之后才执行 fix 会话识别与合规判定，**只有此阶段**的 transcript 缺失/解析失败/超时/内部异常进入 FR-013 的 fail-open 放行。两类回落（配置层→回落默认强度；判定层→fail-open 放行）必须以类型化错误或显式分支区分，**禁止用单个 catch-all 同时接住两类失败**（否则 YAML 损坏会被误当判定异常而 fail-open，整体绕过默认强度语义）。

  评测场景的硬 enforce 由"默认值即 `block`"这一语义保证，机制不得探测评测环境并差异化行为（评测目标 repo 无项目级配置覆盖，天然执行默认强度；已核实 init-project.sh 只检查不创建 config、fix SKILL 配置缺失时仅用内存默认值不写盘）。**[必须]**——去掉可配置性，通用用户场景（F204 教训：通用插件内的强干预机制必须留给用户退路）将被迫接受硬阻断且无合法降级通道，损害插件通用可用性；去掉"默认 block + 禁环境探测"约束，评测硬 enforce 语义失效或机制退化为评测特判（与 FR-011 红线冲突）；去掉判定顺序约束，两类回落语义在实现中可被 catch-all 合并，导致配置损坏静默绕过强制。

### Key Entities *(include if feature involves data)*

- **Fix 会话合规状态**：一次 fix 模式会话在结束时是否满足最低依从性要求的判定结果，包含收口形态（修复收口/no-op 收口，修复收口下再分轻量/完整验证变体）、缺失项清单、判定所依据的客观信号来源。
- **依从性判据清单**：按收口形态（修复收口/no-op 收口）区分的最低要求集合（必需制品及其必填章节、最低委派次数与角色类型、委派内容与判断结论的关联性要求）。
- **阻断记录**：单次会话内每次拦截事件的记录，包含阻断原因、反馈给 agent 的补救指引、是否为达到上限后的降级放行、本次阻断计数所归属的会话标识（用于并发场景下的隔离存储与追溯）。

## 复杂度评估（供 GATE_DESIGN 审查）

- **组件总数**：3 个新增/改造组件——(1) 新增 Stop hook 脚本（fix 会话依从性校验，独立于现有非阻断型 `stop-task-check.sh`）；(2) `record-workflow-run.mjs` 由"只记录"升级为"可选启用主动校验制品清单"（默认行为对现有 5 个 SKILL 调用方不变）；(3) SKILL.md 新增"确认无需改动"专用出口分支（含轻量制品模板）。判定：**3 个，落在 MEDIUM 边界（3-5 区间下限）**。
- **接口数量**：约 4-5 处——Stop hook 的阻断决策接口（读 transcript + 制品目录，输出 block/allow + reason）、`record-workflow-run.mjs` 新增的可选校验参数接口（如必需制品清单、最低委派数、收口形态标识）、SKILL.md 新增出口分支的产物格式约定、阻断次数上限与降级标注的记录格式、`spec-driver.config.yaml` 强制程度配置项（FR-015，block/warn/off 三值）。判定：**4-5 个，落在 MEDIUM 区间**。
- **依赖新引入数**：0——技术调研已明确无新增 npm 依赖，仅使用 Node 内置能力（`node:fs`/`node:path`）与 Claude Code 平台内置 hooks 机制，延续仓库现有实现风格。
- **跨模块耦合**：是——本次改动需要同时修改 hooks 配置/脚本、`record-workflow-run.mjs`、SKILL.md 三处既有模块的行为契约，属于 2+ 个现有模块协同改造（Stop hook 依赖 record-workflow-run 产出的结构化状态作为判据来源，二者是同一机制的两端）。`record-workflow-run.mjs` 现有 5 个 SKILL 调用方（fix/story/implement/doc/resume），本次升级要求对其中 4 个（非 fix）保持零行为变化，是本次跨模块耦合中风险最高的一环（详见 FR-014）。
- **复杂度信号**：存在 1 个——**状态机式的会话生命周期判定**（拦截-反馈-补救-再判定-放行/降级放行的多状态流转，含阻断次数计数与有界化逻辑）。不存在递归结构、不存在真正意义上的并发控制（单会话内的顺序阻断判定不构成并发场景，多会话并发仅要求状态隔离存储而非并发协调）、不存在数据迁移。
- **总体复杂度**：**MEDIUM**（组件数 3 落在 3-5 区间下限，跨模块耦合成立，且存在 1 个复杂度信号——状态机式判定流程，符合"组件 3-5 或存在 1 个复杂度信号"的 MEDIUM 判定规则）。建议 GATE_DESIGN 对 Stop hook 会话自判逻辑、阻断有界化设计与 `record-workflow-run.mjs` 向后兼容边界（FR-014）给予正常强度的人工审查，不需要升级为 HIGH 复杂度的加强审查流程。
