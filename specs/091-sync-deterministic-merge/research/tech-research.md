# Feature 091 — Sync Deterministic Merge 技术调研报告

> **调研模式**: tech-only
> **调研日期**: 2026-04-06
> **调研目标**: 将 sync 子代理的合并算法提取为确定性 MJS 脚本

---

## 1. 架构方案选型

### 方案 A：全量提取 — Merge Engine 单脚本（推荐）

**核心思路**：将 sync.md 中所有可确定性执行的操作提取到 `sync-merge-engine.mjs` 一个入口脚本 + `scripts/lib/` 下多个纯函数模块。Agent Prompt 仅保留语义决策层。

**模块拆分**：

```
scripts/
  sync-merge-engine.mjs          # CLI 入口（--dry-run, --json, --project-root）
  lib/
    sync-product-mapping.mjs     # product-mapping.yaml 读写、产品名修正、差集检测
    sync-timeline-builder.mjs    # 按编号排序 + 类型标记（INITIAL/FEATURE/FIX/REFACTOR/ENHANCEMENT）
    sync-merge-strategy.mjs      # 增量合并策略：按类型的确定性合并规则
    sync-conflict-resolver.mjs   # 冲突解决：编号更大优先
    sync-validator.mjs           # 功能数量检查、矛盾检测、Markdown 结构校验
    sync-changelog.mjs           # 变更历史生成
```

**Agent 保留的语义层**（约 3,000-4,000 bytes）：
- Section 2.2 内容分析推断产品归属（需 LLM 理解 spec 内容）
- Section 4 的 14 章语义融合（综合描述提炼、用户画像推断等）
- 信息推断规则（User Stories -> 用户画像）
- 对外文档摘要生成

**执行流程**：
1. Agent 调用脚本执行确定性预处理（映射加载、修正、差集、排序）
2. 脚本返回结构化 JSON（产品分组 + 时间线 + 合并骨架）
3. Agent 接收 JSON，执行语义融合，生成 current-spec.md
4. Agent 调用脚本执行后验证

**优势**：
- Agent Prompt 可从当前约 8,000+ bytes 缩至 <5,000 bytes
- 确定性操作 100% 可测试、可复现
- 脚本 `--dry-run` 可独立预览合并计划

**劣势**：
- 脚本需要理解 spec.md 的 Markdown 结构以提取结构化字段
- 初始开发工作量较大

**推荐理由**：完全符合宪法原则 III（YAGNI——只做当前 sync 需要的）和原则 X（零运行时依赖）。与现有 `generate-product-entity-catalog.mjs` 风格一致。

---

### 方案 B：渐进提取 — 分阶段外移

**核心思路**：不一次性提取全部确定性逻辑，而是分三批逐步外移。

**Phase 1**（产品映射层）：
- 只提取 Section 2.1/2.3/2.4（映射读取、名称修正、差集检测）
- 脚本约 200 行，Agent Prompt 减少约 1,500 bytes

**Phase 2**（时间线 + 合并策略层）：
- 提取 Section 3a/3b/3c（构建时间线、合并策略、冲突解决）
- 脚本追加约 300 行

**Phase 3**（验证 + 变更历史层）：
- 提取 Section 5 和 Section 4.12
- 脚本追加约 150 行

**优势**：
- 每次改动范围小，风险可控
- 可在任意 Phase 暂停，已外移部分立即生效

**劣势**：
- 三次 Prompt 修改 + 三次 Gate 验证，开发周期更长
- Phase 间可能出现接口设计返工
- 中间状态的 Prompt 仍然偏大

---

### 方案 C：纯 Prompt 瘦身 — 不引入脚本

**核心思路**：将当前 Prompt 中的伪代码重写为更紧凑的结构化指令，不引入外部脚本。

**优势**：
- 零代码改动，纯文本编辑
- 无运行时变更风险

**劣势**：
- Prompt 瘦身空间有限（估计只能从 8K 降到 6K，难以达到 <5K 目标）
- 合并逻辑仍不可测试、不可复现
- 无法提供 `--dry-run` 预览

**结论**：不满足需求目标，不推荐。

---

### 选型推荐

**推荐方案 A**。理由：
1. 一步到位，避免方案 B 的多次 Prompt 迭代
2. 与现有脚本架构完全对齐（参见第 5 节风格分析）
3. 脚本模块化程度高，每个 `lib/sync-*.mjs` 都是纯函数，可独立测试
4. YAGNI 合规：只提取当前 sync.md 已有的确定性操作，不扩展功能

## 2. 业界工具调研结论

### 2.1 文档合并/聚合工具

| 工具 | 聚合方式 | 与本场景的关联度 | 可借鉴点 |
|------|---------|----------------|---------|
| **mdbook** | SUMMARY.md 定义章节顺序和层级，严格的目录结构驱动。`mdbook-autosummary` 插件可从文件系统自动生成 SUMMARY.md | 中 | SUMMARY.md 作为"合并清单"的思路可借鉴：`product-mapping.yaml` 类似 SUMMARY.md，定义了哪些 spec 参与合并 |
| **Docusaurus** | sidebar 从文件系统自动生成，支持 `_category_.json` 元数据覆盖，可拼接多个 autogenerated 片段 | 中 | 文件名数字前缀排序 + 元数据覆盖的优先级链与本场景的"编号排序 + 映射覆盖"一致 |
| **unified.js / remark** | Markdown -> mdast AST -> 变换 -> Markdown。多文档合并通过 `{ type: 'root', children: [...doc1.children, ...doc2.children] }` 实现 | 高（模式可借鉴，但受宪法原则 X 限制不能直接用） | AST 层合并的思路值得借鉴：将 spec.md 解析为结构化数据再合并，而非字符串拼接 |
| **Hugo / Gatsby** | 内容管道（content pipeline）：source -> transform -> aggregate -> render。Hugo 用 Go template 聚合，Gatsby 用 GraphQL 查询层 | 低 | pipeline 阶段划分思路可参考，但复杂度远超本场景需求 |

**关键借鉴**：
- 采用 unified.js 的 **"解析→AST→变换→序列化"** 四阶段模型，但不引入 npm 包
- 自行实现轻量 Markdown section parser（提取 H2/H3 级别的章节结构），类似现有 `simple-yaml.mjs` 的定位
- 合并清单（product-mapping.yaml）沿用 mdbook SUMMARY.md 的模式

### 2.2 本场景特殊性

与通用文档合并工具相比，本场景有两个独特约束：
1. **语义层不可消除**：14 章中部分内容需要 LLM 综合推理（如"提炼产品定位"），纯 AST 合并无法覆盖
2. **零依赖约束**：宪法原则 X 禁止引入 npm 包，必须自实现所有解析/合并逻辑

因此本场景的正确定位是 **"结构化预处理 + LLM 语义融合"** 的 hybrid 模式，而非纯工具链。

## 3. LLM + 确定性工具分工模式分析

### 3.1 业界模式对比

| 项目/模式 | LLM 职责 | 工具职责 | 分工界面 | 与本场景的契合度 |
|-----------|---------|---------|---------|----------------|
| **Aider** Architect/Editor | Architect 模型生成编辑意图（plain text 指令），Editor 模型/工具执行 apply | search/replace block 或 unified diff 格式 apply | LLM 输出结构化编辑指令 -> 确定性 apply | 高：Agent 输出"合并决策"，脚本执行"合并操作" |
| **Cursor** 两阶段模型 | 主模型生成代码"草图"（sketch），不要求完美格式 | Apply 模型（训练过的小模型）将 sketch 集成到文件 | 主模型 -> sketch -> Apply 模型 -> 文件 | 中：本场景不需要第二个模型，确定性脚本即可 |
| **LangGraph** Plan-then-Execute | Planner LLM 生成结构化计划（graph traversal plan） | Executor 按计划逐步执行工具调用 | JSON plan -> 确定性执行 | 高：与方案 A 的 "Agent 决策 -> 脚本执行" 完全对齐 |
| **Terraform** plan/apply | 用户/CI 定义声明式配置 | `terraform plan` 确定性计算 diff，`terraform apply` 执行 | plan（预览）-> 确认 -> apply（执行） | 非常高：`--dry-run` 直接对应 `terraform plan` |

### 3.2 推荐分工模式

采用 **LangGraph 风格的 Plan-then-Execute**，具体映射为：

```
┌──────────────────────┐     JSON      ┌──────────────────────┐
│  sync Agent (LLM)    │ ──────────→  │  sync-merge-engine   │
│                      │              │  (MJS 脚本)           │
│  - 产品归属推断       │  ←────────── │                      │
│  - 语义融合 (14章)    │   结构化结果  │  - 映射读写           │
│  - 信息推断规则       │              │  - 时间线构建          │
│  - 摘要生成          │              │  - 确定性合并          │
│                      │              │  - 冲突解决            │
│                      │              │  - 验证                │
└──────────────────────┘              └──────────────────────┘
```

**分工原则**：
- **确定性 = 脚本**：输入确定则输出确定的操作（排序、匹配、差集、格式校验）
- **语义性 = Agent**：需要理解自然语言含义的操作（归属推断、内容综合、摘要生成）
- **接口 = JSON**：Agent 与脚本之间通过 JSON 传递数据，避免 Prompt 中嵌入结构化数据

### 3.3 Aider 经验的关键借鉴

Aider 的演化路径揭示了一个重要教训：LLM 生成精确 diff 的可靠性很低（约 60% 成功率），因此 Aider 从 diff 格式转向了 search/replace block 格式。对本场景的启示是：

- **不要让 Agent 输出结构化合并指令**：Agent 应输出语义级决策（"spec-042 属于 reverse-spec 产品"），而非操作级指令（"在 product-mapping.yaml 第 15 行追加条目"）
- **脚本消费决策，自主执行操作**：脚本接收高层决策，自行计算具体的文件操作

## 4. --dry-run 输出格式推荐

### 4.1 格式对比

| 格式 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **diff 风格**（+/- 行标记） | 开发者熟悉；git diff 工具链兼容 | 对新生成文件（无旧版本）无意义；Markdown 内容的 diff 可读性差 | 增量更新已有 current-spec.md |
| **预览 Markdown** | 所见即所得；可直接验证最终产物 | 输出量大（完整文件）；无法快速定位变更 | 首次生成 current-spec.md |
| **JSON changelog** | 机器可读；可集成到 CI；结构紧凑 | 人类不直观；需要工具渲染 | 自动化流水线消费 |
| **混合格式**（summary + 关键 diff） | 兼顾可读性和信息密度；Terraform plan 已验证此模式 | 实现复杂度稍高 | 通用场景（推荐） |

### 4.2 推荐：混合格式（借鉴 Terraform plan）

Terraform 的 `plan` 输出是 dry-run 的黄金标准——先给出摘要统计，再展示具体变更。建议采用类似结构：

```
========================================
  Sync Merge Engine — Dry Run Preview
========================================

产品映射变更:
  ~ spec-driver: 新增 2 个 spec (091, 092)
  (无删除)

产品: spec-driver
  时间线: 45 个 spec (INITIAL: 1, FEATURE: 32, FIX: 6, REFACTOR: 3, ENHANCEMENT: 3)
  合并骨架:
    第5章 当前功能全集: +3 FR (来自 091, 092), ~1 FR 更新 (来自 090-fix)
    第11章 被废弃的功能: +1 (来自 089-refactor)
    第12章 变更历史: +3 条目
  冲突: 0

产品: reverse-spec
  时间线: 12 个 spec (INITIAL: 1, FEATURE: 9, FIX: 2)
  合并骨架: 无变更

验证: 全部通过 (3/3 检查)

摘要: 2 个产品, 57 个 spec, 0 个冲突, 0 个警告
```

当使用 `--json` 标志时，输出完整的 JSON 结构以供脚本消费：

```json
{
  "dryRun": true,
  "products": {
    "spec-driver": {
      "timeline": { "total": 45, "byType": { "INITIAL": 1, "FEATURE": 32, ... } },
      "mappingChanges": { "added": ["091", "092"], "removed": [] },
      "mergePreview": {
        "chapters": {
          "5": { "addedFR": 3, "updatedFR": 1, "sources": ["091", "092", "090-fix"] },
          ...
        }
      },
      "conflicts": [],
      "validationResults": [...]
    }
  },
  "summary": { "totalProducts": 2, "totalSpecs": 57, "conflicts": 0, "warnings": 0 }
}
```

### 4.3 设计决策

- 默认输出人类可读的混合格式
- `--json` 输出 machine-readable JSON
- 不实现 diff 风格：current-spec.md 由 Agent 语义融合生成，脚本层面无法产出有意义的逐行 diff
- 不输出完整预览 Markdown：current-spec.md 最终由 Agent 生成，脚本只能预览"合并骨架"

## 5. 现有脚本风格分析

### 5.1 模块化模式

通过分析 `generate-product-entity-catalog.mjs` 及 `scripts/lib/` 下的模块，总结出以下稳定模式：

**入口脚本模式**：
```javascript
#!/usr/bin/env node
import fs from 'node:fs';
import path from 'node:path';
import process from 'node:process';
// lib 模块导入...

// 1. 内部 parseArgs() 函数解析 CLI 参数
function parseArgs(argv) { ... }

// 2. 导出核心逻辑函数（支持 programmatic 调用）
export function mainLogic(options = {}) { ... }

// 3. 尾部 CLI 入口守卫
if (import.meta.url === `file://${process.argv[1]}`) {
  const args = parseArgs(process.argv.slice(2));
  const result = mainLogic(args);
  printResult(result, args.json);
}
```

**CLI 参数约定**：
- `--project-root <path>`：项目根目录（默认 `process.cwd()`）
- `--json`：JSON 格式输出
- `parseCommonProjectArgs()` 提供通用参数解析（`script-cli-args.mjs`）
- 新增 `--dry-run` 需自行扩展

**lib 模块约定**：
- 每个模块只 export 纯函数或工具函数
- 不做 side effect（不读写文件），文件 I/O 在入口脚本中完成
- 模块间通过 import 显式依赖
- `simple-yaml.mjs`：轻量 YAML 解析/序列化（纯手写，零依赖）
- `script-report-io.mjs`：统一的文件写入工具（`writeJsonArtifact`, `writeYamlArtifact`, `writeMarkdownArtifact`）
- `product-artifact-paths.mjs`：路径集中管理
- `product-governance-helpers.mjs`：通用工具函数（`parseProductMapping`, `slugToTitle`, `isObject`）
- `script-diagnostics.mjs`：诊断辅助（`dedupeStringValues`, `appendWarningsSection`）

### 5.2 错误处理模式

- 缺少关键文件：`throw new Error('未找到 ...: ${path}')` 中文错误信息
- 可选文件缺失：返回 `null` 或默认值，不中断流程
- try/catch 限于外部命令（如 `execFileSync`），内部逻辑通过提前检查避免异常
- 警告收集在 `warnings` 数组中，最终一次性输出

### 5.3 代码风格

- 严格 ES Module（`.mjs` 后缀）
- 只用 Node.js 内置模块（`fs`, `path`, `process`, `child_process`）
- 函数命名：驼峰式英文，中文仅出现在注释和错误信息中
- 常量用 `UPPER_SNAKE_CASE`
- 对象判断统一使用 `isObject()` helper
- path 统一转 POSIX（`toPosix()`）

### 5.4 对 sync-merge-engine.mjs 的设计启示

- 入口脚本沿用 `import.meta.url === file://...` 守卫 + `export function` 双入口模式
- CLI 增加 `--dry-run` 参数（扩展 `parseCommonProjectArgs` 或自行实现 `parseArgs`）
- lib 模块遵循纯函数 + 零副作用原则
- 复用现有 `simple-yaml.mjs`（product-mapping.yaml 解析）、`product-artifact-paths.mjs`（路径管理）、`script-report-io.mjs`（文件写入）
- 新增的 `sync-*.mjs` 模块与现有 `product-*.mjs` 命名风格一致

## 6. 技术风险清单

| # | 风险 | 影响 | 概率 | 缓解措施 |
|---|------|------|------|---------|
| R1 | **spec.md Markdown 结构多样性**：不同 spec 的 Markdown 格式不统一（标题层级、列表格式等），导致脚本解析失败 | 高 | 中 | 实现宽松的 section parser，以 H2（`##`）为主要分割点；解析失败时返回 raw text 而非抛异常；参考 `parseCurrentSpec()` 的容错模式 |
| R2 | **确定性/语义边界模糊**：某些操作看似确定性但实际需要语义判断（如"FR 是否重复"的判定） | 中 | 中 | 初始版本保守划界——只提取确定性无争议的操作；模糊操作留给 Agent；后续可逐步外移 |
| R3 | **降级兼容性**（宪法原则 XIII）：脚本不存在或执行出错时 sync 流程需要合理降级 | 高 | 低 | Agent Prompt 中保留降级路径："若脚本不可用，按以下简化规则执行"（约增加 500 bytes）；脚本执行失败返回明确错误码 |
| R4 | **Agent 与脚本的接口契约漂移**：脚本输出 JSON 格式变更后 Agent Prompt 未同步更新 | 中 | 中 | 在脚本输出 JSON 中包含 `schemaVersion` 字段（参考 `entity.yaml` 的模式）；Agent Prompt 中声明期望的 schema version |
| R5 | **合并骨架与语义融合的脱节**：脚本生成的合并骨架与 Agent 最终输出的 current-spec.md 在结构上不一致 | 中 | 低 | 脚本输出的骨架采用与 `product-spec-template.md` 相同的 14 章结构；Agent 基于骨架填充内容而非重建结构 |
| R6 | **性能风险**：spec 数量增长后脚本执行时间过长 | 低 | 低 | 当前 spec 数量约 90 个，纯文件读取 + 排序 + 合并在 Node.js 下应 <1s；暂不需要优化 |
| R7 | **product-mapping.yaml 写入冲突**：脚本写入 mapping 文件时与 Agent 并发操作 | 中 | 低 | 明确执行顺序：脚本先写 mapping -> Agent 读取 -> Agent 后续操作。不支持并发写入 |
